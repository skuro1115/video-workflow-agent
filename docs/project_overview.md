# Project Overview

## 何を作るのか

長尺動画（1時間級の配信アーカイブやイベント収録など）から、面白い・使えそうな箇所を検出して短尺クリップに加工する **自動化パイプライン** を作る。

「人間の編集者が、まず通しで観て、目立った場面にマーカーを置き、そこを切り出して短く整える」という作業を、

1. 機械的なメタデータ抽出
2. 検出器による候補抽出
3. 計画 → 切り出し
4. （将来）字幕・サムネイル・SNS 投稿

の各段階に分割して自動化する。

## なぜ段階分割するのか

- **AI 解析は高コスト** — 1時間動画にいきなり LLM をかけると遅い・高い・再現性が低い。先に安価な信号（音声 RMS・シーン検出・フレーム差分）で粗く絞ってから精度の高い解析を当てる方が現実的
- **検出ロジックは絶対に差し替わる** — 最初の検出器がベストである確率はほぼゼロ。インタフェースで切れる構造にしておかないと、後から差し替えるたびに大改修になる
- **「なぜそこを切ったのか」が後から必要になる** — 短尺クリップを人間がレビューするとき、検出器の意図が分からないと判断できない。`reason` / `score` を全段で持ち回す

## スコープ（MVP）

| 含む | 含まない |
| --- | --- |
| ローカルファイル入力 | YouTube / 配信プラットフォーム直接取り込み |
| 単一動画の処理 | 複数動画のバッチ・キュー処理 |
| ffmpeg / ffprobe ベースの処理 | クラウド実行・分散処理 |
| JSON でのメタデータ出力 | データベース永続化 |
| CLI 実行 | Web UI / 常駐サービス |
| プレースホルダ検出器 | 実用的な盛り上がり検出 |

MVP の目的は **「パイプラインの両端が繋がっている状態」** を作ること。検出器の精度はそのあとで上げる。

## 想定する最終形

```
[入力]            [段階1]               [段階2]            [段階3]
1時間動画 ─→ ホットポイント候補 ─→ 10分ダイジェスト ─→ 10〜60秒クリップ × N
              （音声 / シーン /         （要約・整形）    （SNS 配信用）
               字幕 / LLM 統合）
```

各段階の出力は JSON で永続化され、人間が中断・再開・修正できる。

## 非ゴール（少なくとも当面）

- 自動投稿の完全自動化（必ず人間レビューを挟む前提）
- 動画生成 AI による補完・差し替え
- ライブ配信への適用（オフライン処理に限定）
